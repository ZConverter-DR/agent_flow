# 프로젝트 배경 지식 정리

이 프로젝트는 **ZIASTACK(OpenStack 기반 환경)에서 AI 에이전트를 이용해 운영 작업, 특히 DR(재해복구) 시나리오를 자동화하려는 프로젝트**로 보인다.

현재 문서와 코드를 함께 보면, 이 프로젝트는 크게 두 층위로 이해하는 것이 맞다.

1. **기획 의도**
2. **현재 실제 구현 상태**

---

## 1. 기획 의도

문서 기준으로 이 프로젝트의 최종 목표는 다음과 같다.

- 운영자가 UI에서 복구 대상을 선택하거나 입력한다.
- AI 에이전트가 OpenStack 상태를 조회한다.
- RAG로 런북, 복구 이력, 장애 대응 지식을 검색한다.
- 검색 결과와 인프라 상태를 바탕으로 복구 정책을 생성한다.
- 운영자가 정책을 승인 또는 거절한다.
- 승인되면 MCP Tool Calling을 통해 OpenStack 복구 작업을 자동 실행한다.
- 복구 완료 후 결과 리포트를 생성하고 저장/알림 전파까지 수행한다.

즉, 사용자가 이해한
“**client가 채팅 UI에 프롬프트를 입력하면, 이를 RAG로 처리하고, tool calling을 호출하고, 리포트를 생성하여 저장하는 목적**”
이라는 설명은 **대체로 맞다.**

다만 더 정확히 말하면, 이 프로젝트의 중심 유스케이스는 일반적인 자유형 채팅 서비스라기보다:

- **운영자용 AI 작업 보조/실행 시스템**
- 그중에서도 **OpenStack DR 복구 자동화**

에 가깝다.

---

## 2. 현재 실제 구현 상태

현재 저장소에 들어 있는 코드는 위 기획 전체를 완성한 상태는 아니다.
오히려 **“AI 채팅 게이트웨이의 골격 + MCP 기반 Tool 연결 실험” 단계**에 가깝다.

### 현재 존재하는 주요 구성요소

#### 2-1. `agent_server/`

FastAPI 기반 서버이며, 사실상 현재 프로젝트의 핵심 실행부다.

- WebSocket 채팅 엔드포인트를 제공한다.
- JWT를 검증한다.
- LangGraph Supervisor 기반 에이전트를 실행한다.
- MCP 서버에 연결해 도구를 호출할 수 있다.

즉, 클라이언트가 WebSocket으로 메시지를 보내면 이 서버가 AI 에이전트를 통해 응답을 생성하는 구조다.

#### 2-2. 인증 구조

코드상 인증은 Horizon/Django 쪽이 JWT를 발급하고, FastAPI가 이를 검증하는 모델로 설계되어 있다.

- JWT 서명 검증
- issuer / audience 검증
- 만료 시간 검증
- Redis 기반 `jti` replay 방지
- Redis 기반 세션 존재 여부 확인

즉, 단순 챗봇이 아니라 **기존 운영 포털(Horizon)과 연결되는 내부 운영용 시스템**으로 설계되어 있다.

#### 2-3. MCP Tool Calling

현재 에이전트는 여러 MCP 도구를 붙일 수 있게 되어 있다.

- Slack MCP
- Filesystem MCP
- OpenStack MCP

그리고 Supervisor가 사용자 요청을 보고 어느 에이전트에게 위임할지 결정한다.

즉, 이 프로젝트는 단순 RAG 챗봇이 아니라
**“채팅 인터페이스를 통해 여러 운영 도구를 호출하는 AI 오케스트레이터”**
방향으로 구현되고 있다.

---

## 3. OpenStack / DR 관점에서 이해해야 할 점

문서 기준으로 이 프로젝트의 핵심 도메인은 분명히 **DR 복구 자동화**다.

주요 흐름은 다음과 같다.

1. 운영자가 복구 요청
2. AI가 OpenStack 리소스/타겟 VM 상태 확인
3. RAG로 런북/이력 검색
4. 복구 정책 생성
5. 운영자 승인
6. MCP를 통해 VM 생성 또는 복구 실행
7. 상태 스트리밍
8. 결과 리포트 작성 및 저장

즉, 채팅 UI는 목적 그 자체라기보다
**운영자가 AI에게 복구 작업을 요청하고 승인하는 인터페이스**에 더 가깝다.

## 4. 현재 코드에서 실제로 되는 것과 안 되는 것

### 현재 실제로 되는 것

- FastAPI 서버 기동
- WebSocket 채팅 연결
- JWT 검증
- Redis 연동
- LangGraph Supervisor 기반 에이전트 실행
- MCP를 통한 Slack / Filesystem / OpenStack 도구 연결 구조
- OpenStack 관련 기본 도구 정의
  - 서버 정보 조회
  - VM 생성
  - 복구 실행
  - 복구 상태 조회

### 아직 실제 구현이 부족하거나 미완성인 것

- **RAG 실구현이 없음**
  - `app/knowledge/runbooks/`는 비어 있음
  - Chroma 인덱싱/검색 파이프라인 없음
- **리포트 저장 구조가 미완성**
  - DB 모델/영구 저장 로직 없음
- **실제 OpenStack 연동이 아님**
  - 현재 OpenStack MCP 핸들러는 mock 데이터/가짜 job store 기반
- **DR 승인 루프가 미완성**
  - 승인/거절/HITL 플로우 없음
- **대화 이력 관리 미완성**
  - Redis history 저장 코드가 주석 처리됨
- **장시간 작업 상태 관리 미완성**
  - job 상태 저장/재시도/blocked 처리 미흡
- **문서상 Claude API 중심 설계와 실제 코드가 다름**
  - 문서에는 Claude API 중심으로 적혀 있지만
  - 현재 코드의 LLM은 `ChatOpenAI` 인터페이스로 `qwen2.5:7b` + Ollama endpoint를 사용 중

즉, “RAG + Tool Calling + Report 저장”이라는 개념은 문서상 목표에는 포함되지만,
**현재 코드 기준으로는 Tool Calling 골격만 어느 정도 있고, RAG와 리포트 저장은 아직 거의 구현되지 않았다**고 보는 것이 정확하다.

---

## 5. 이 프로젝트를 한 문장으로 요약하면

이 프로젝트는
**“Horizon/OpenStack 운영 환경에서, AI 에이전트가 채팅 인터페이스를 통해 인프라 정보를 조회하고 도구를 호출하며, 장기적으로는 DR 복구 정책 생성·실행·보고까지 자동화하려는 시스템”**
이라고 정리할 수 있다.

---

## 6. 당신이 지금 이해하고 있으면 좋은 핵심 포인트

- 이 프로젝트는 일반 사용자용 챗봇이 아니다.
- 핵심 도메인은 OpenStack 기반 운영 자동화, 특히 DR 복구다.
- 채팅 UI는 에이전트와 상호작용하는 프론트엔드일 뿐이다.
- Tool Calling은 현재 가장 구체적으로 구현된 축이다.
- RAG와 리포트 저장은 “목표”에 가깝고, 현재 구현은 미완성이다.
- 문서상 이상적인 아키텍처와 현재 코드 구현 사이에 차이가 있다.

---

## 7. 결론

당신이 알고 있던 설명은 방향성 면에서는 맞다.
다만 현재 프로젝트를 더 정확히 표현하면 다음과 같다.

> 이 프로젝트는 채팅 UI를 통해 AI 에이전트가 운영 작업을 수행하도록 만드는 시스템이며,
> 그 주요 목표는 OpenStack DR 복구 자동화이다.
> RAG, Tool Calling, 리포트 생성/저장은 모두 목표 범위에 포함되지만,
> 현재 코드 기준으로는 Tool Calling 중심의 초기 골격이 구현되어 있고
> RAG/리포트 저장/완전한 복구 워크플로우는 아직 미완성 상태다.
