## 전체적으로 달라진 점

기존 코드는 `agent_server/main.py` 안에서 MCP tool 로딩, Redis checkpointer 생성, Agent 생성, WebSocket 라우터 등록이 한 번에 섞여 있었고, 채팅 WebSocket도 `app/ws/chat.py` 한 파일에 운영용 인증과 메시지 루프가 함께 들어 있었다.

현재 코드는 다음처럼 책임이 분리되었다.

- 운영용 채팅과 개발용 채팅 경로가 분리됨
- 인증 처리와 채팅 루프가 분리됨
- 개발용 브라우저 채팅 UI가 추가됨
- MCP/Agent 초기화가 bootstrap 계층으로 분리됨
- dev/prod 모드에 따라 로딩할 MCP와 tool 구성이 달라짐
- RAG를 바로 구현하지는 않았지만, 이후 붙일 수 있는 knowledge 골격과 tool 주입 지점이 추가됨

## Phase 1 기준 변경사항

### 1. WebSocket 구조 분리

기존:

- `agent_server/app/ws/chat.py` 하나가 `/ws/chat` 라우팅, JWT 인증, history 복원, 메시지 루프, confirm 처리까지 모두 담당

현재:

- `agent_server/app/ws/chat_common.py`
  - history 복원
  - 메시지 파싱
  - `confirm_response` 처리
  - Agent 호출
  - confirm 이벤트 전송
- `agent_server/app/ws/chat_prod.py`
  - 운영용 `/ws/chat`
  - Horizon JWT 기반 identity 생성 후 공통 루프 호출
- `agent_server/app/ws/chat_dev.py`
  - 개발용 `/ws/dev-chat`
  - `?user=` 기반 dev identity 생성 후 공통 루프 호출
- `agent_server/app/ws/chat.py`
  - 기존 import 경로 호환용으로 `chat_prod.router`만 다시 export

즉, 기존 단일 파일 구조가 "운영 라우터 / 개발 라우터 / 공통 루프" 3단 분리 구조로 바뀌었다.

### 2. 개발용 identity 추가

기존:

- JWT 검증 성공 결과의 `sub`를 그대로 `thread_id`로 사용
- 개발용 identity 개념 자체가 없음

현재:

- `agent_server/app/auth/models.py`
  - `ChatIdentity(auth_mode, thread_id, username, subject)` 추가
- `agent_server/app/auth/providers/dev_identity.py`
  - `user` query parameter를 받아 dev identity 생성
  - 값이 비어 있으면 `WS_1008_POLICY_VIOLATION`으로 거부
  - `thread_id`를 `dev:{username}` 형태로 고정
  - `auth_mode="dev"`로 구분 가능

즉, 개발용 WebSocket은 JWT를 흉내 내지 않고 별도 identity 모델을 사용하게 되었다.

### 3. 운영용 인증 분리

기존:

- `app/ws/chat.py` 안에서 직접 `verify_jwt()` 호출

현재:

- `agent_server/app/auth/providers/horizon_jwt.py`
  - Redis 획득 후 `verify_jwt()`를 호출해 `ChatIdentity`로 변환
- `agent_server/app/ws/chat_prod.py`
  - 운영용 라우터에서 해당 provider를 사용

즉, JWT 검증 의미는 유지하면서도 운영용 인증 책임이 WebSocket 루프 코드에서 분리되었다.

### 4. 개발용 채팅 UI 추가

기존:

- FastAPI가 직접 제공하는 개발용 채팅 페이지가 없었음
- 브라우저에서 dev용으로 바로 접속해 테스트할 UI가 없었음

현재:

- `agent_server/app/devui/routes.py`
  - `GET /dev/chat` HTML 페이지 추가
- `agent_server/app/devui/static/dev-chat.js`
  - WebSocket 연결, 메시지 송수신, history/confirm 처리
- `agent_server/app/devui/static/dev-chat.css`
  - 개발용 채팅 페이지 스타일

UI에는 다음 요소가 들어갔다.

- 사용자명 입력
- 연결 버튼
- 메시지 입력/전송
- 대화 로그 영역
- 연결 상태 표시
- confirm approve/reject 버튼

### 5. 개발용 기능 활성화 조건 추가

기존:

- dev chat 기능을 켜고 끄는 설정이 없었음

현재:

- `agent_server/app/common/config.py`
  - `app_mode`
  - `dev_chat_enabled`
  - 관련 helper property 추가
- `agent_server/main.py`
  - `settings.dev_chat_enabled`가 `true`일 때만
    - `/ws/dev-chat`
    - `/dev/chat`
    - `/dev/static`
    를 등록

즉, 개발용 기능이 상시 노출되지 않고 설정 기반으로만 활성화되도록 바뀌었다.

## Phase 2 기준 변경사항

### 1. Agent 초기화 책임 분리

기존:

- `agent_server/main.py` lifespan에서
  - Slack MCP client 생성
  - OpenStack MCP client 생성
  - tool 로딩
  - Redis checkpointer 생성
  - Agent 생성
  를 모두 직접 수행

현재:

- `agent_server/app/bootstrap/tools.py`
  - 설정 기반 MCP 로딩
- `agent_server/app/bootstrap/checkpointer.py`
  - Redis checkpointer 생성
- `agent_server/app/bootstrap/agent.py`
  - startup 로그 출력
  - tool 로딩
  - 필요 시 knowledge tool 병합
  - 최종 Agent 생성
- `agent_server/main.py`
  - 위 bootstrap 결과만 받아 app state에 저장

즉, main은 조립 지점만 맡고 실제 초기화 로직은 bootstrap 계층으로 빠졌다.

### 2. 설정 기반 MCP on/off

기존:

- Slack MCP와 OpenStack MCP를 사실상 고정으로 로딩
- mock MCP는 주석 상태

현재:

- `agent_server/app/common/config.py`
  - `enable_slack_mcp`
  - `enable_openstack_mcp`
  - `enable_test_mock_mcp`
  - `is_dev_mode`
  - 각 MCP의 기본 활성화 정책 helper 추가
- 기본 정책:
  - dev 모드: Slack/OpenStack off, Mock on
  - prod 모드: Slack/OpenStack on, Mock off

즉, dev 환경에서 OpenStack/Slack이 없어도 앱이 뜰 수 있는 구조가 생겼다.

### 3. Optional MCP 로딩 정책 반영

기존:

- MCP 로딩 실패 시 startup 단계 전체가 깨질 가능성이 컸음

현재:

- `app/bootstrap/tools.py`
  - 활성화된 MCP만 순서대로 로드
  - dev 모드에서는 MCP 로딩 실패 시 warning 로그 후 계속 진행
  - prod 모드에서는 실패를 예외로 올려 startup 실패 허용

즉, 개발 모드에서 외부 의존성 때문에 서버 전체가 반드시 죽지 않도록 바뀌었다.

### 4. Agent tool 구성 방식 변경

기존:

- `build_agent(slack_tools, openstack_tools, checkpointer)` 시그니처
- 최종 tool 목록은 사실상 `slack_tools + openstack_tools`

현재:

- `build_agent(tools, checkpointer, extra_tools=None)` 시그니처로 변경
- bootstrap에서 구성한 tool 목록을 일괄 전달
- 필요 시 `extra_tools`를 통해 RAG/knowledge tool 추가 가능
- tool이 하나도 없으면 startup warning 로그를 남김

즉, Agent가 특정 MCP 조합에 고정되지 않고 설정 기반 조합을 받을 수 있게 되었다.

### 5. startup 로그 개선

기존:

- 어떤 모드로 떴는지, 어떤 MCP가 실제 로드됐는지 startup 로그만으로 파악하기 어려웠음

현재:

- `app/bootstrap/agent.py`에서 startup 시 다음을 로그로 남김
  - `APP_MODE`
  - `DEV_CHAT_ENABLED`
  - `ENABLE_RAG`
  - MCP별 활성화 여부
  - 실제 활성 MCP 목록
  - 최종 tool 개수/이름

## Phase 3 기준 변경사항

### 1. RAG 골격 추가

기존:

- knowledge/runbook 관련 구조가 없었음

현재:

- `agent_server/app/knowledge/runbooks/loader.py`
  - `given_docs`에서 `.md`, `.txt` 문서를 읽는 로더
- `agent_server/app/knowledge/runbooks/indexer.py`
  - placeholder index 구조
- `agent_server/app/knowledge/runbooks/retriever.py`
  - 단순 문자열 포함 여부 기반 placeholder retriever
- `agent_server/app/knowledge/tools.py`
  - `search_runbooks` tool 생성

즉, 실제 벡터 검색은 없지만 "문서 로딩 -> 인덱스 -> retriever -> tool" 골격이 준비되었다.

### 2. RAG 설정 추가

기존:

- RAG on/off 설정이 없었음

현재:

- `agent_server/app/common/config.py`
  - `enable_rag` 추가

### 3. Agent 주입 지점 추가

기존:

- Agent 생성 시 knowledge 계층을 끼워 넣을 자리가 없었음

현재:

- `agent_server/app/bootstrap/agent.py`
  - `settings.enable_rag`가 true일 때 `create_knowledge_tools()` 호출
- `agent_server/app/agent/agent.py`
  - `build_agent(..., extra_tools=...)` 형태로 추가 tool 병합 가능

즉, RAG를 완성하지는 않았지만 향후 retriever/tool을 Agent에 붙일 수 있는 연결점이 생겼다.

### 4. 코드 차원 문서화

현재 코드에는 주석으로 다음 의도가 남아 있다.

- runbook 문서는 `given_docs`에서 읽도록 설계됨
- 실제 retriever 생성은 `create_runbook_retriever()`에서 담당
- bootstrap이 `ENABLE_RAG=true`일 때 knowledge tool을 Agent에 합치는 구조

## 기존 코드 대비 핵심 요약

가장 큰 차이는 "운영용 단일 채팅 서버" 구조에서 "운영/개발 분리 + 설정 기반 조립 + 향후 RAG 확장 가능" 구조로 바뀐 점이다.

정리하면:

- 채팅 경로가 운영용 `/ws/chat`과 개발용 `/ws/dev-chat`으로 분리됨
- 개발용 브라우저 UI `/dev/chat`이 새로 생김
- JWT 인증과 dev identity가 provider 계층으로 분리됨
- 공통 메시지 루프가 재사용 가능하게 분리됨
- MCP/tool 로딩이 main에서 bootstrap 계층으로 빠짐
- dev/prod 환경에 따라 외부 의존성을 다르게 로딩할 수 있게 됨
- RAG는 미완성이지만 knowledge 골격과 Agent 주입 포인트가 생김
